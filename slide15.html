第2層 - プロセス空間の事前構造化

Skills化による熟練ノウハウの構造化・継承

⚙️ Skills化とは

特定工程の実行手順・判断基準・成果物テンプレートを文書化し、
生成AIに注入可能な構造化文書として外部化する手法
※ SE/SWEにおける「プロセス」と本質的に同じ概念
📋
3項組(I,S,O)による形式化
// Skills化の数学的定義 Skill := (Input, Steps, Output) Input := 前工程からの成果物 + 制約条件 + 要求仕様 Steps := 実行手順 + 判断基準 + 品質チェック + 検証方法 Output := 次工程への成果物 + 検証結果 + 品質メトリクス
📥 Input（入力）
  • • 前工程の成果物（設計書・仕様書等）
  • • ハードウェア制約（pre-ACD）
  • • 品質要求・性能要求
  • • 利用可能リソース情報
⚙️ Steps（手順）
  • • 段階的実行ステップの定義
  • • 各段階での判断基準・閾値
  • • 品質チェックポイント
  • • エラー処理・例外対応手順
📤 Output（出力）
  • • 次工程への標準化成果物
  • • 検証・テスト結果
  • • 品質メトリクス・KPI
  • • トレーサビリティ情報
🛠️
詳細設計Skillの実装例
// 詳細設計Skill定義例 DetailedDesign_Skill := { Input: { basic_design_document, // 基本設計書 hw_constraints_preACD, // HW制約(pre-ACD) performance_requirements, // 性能要求 safety_requirements // 安全要求 }, Steps: { 1. "モジュール分割" → { method: "cohesion/coupling分析", criteria: "cohesion≥0.8, coupling≤0.3", tools: ["UML", "dependency_analyzer"] }, 2. "API設計" → { method: "contract-first design", criteria: "type-safe, error-handling完備", validation: "OpenAPI仕様生成" }, 3. "実装方針決定" → { method: "design pattern適用", criteria: "保守性・拡張性・性能バランス", review: "アーキテクチャレビュー実施" } }, Output: { detailed_design_document, // 詳細設計書 implementation_guidelines, // 実装ガイドライン api_specification, // API仕様書 quality_metrics // 品質指標 } }
🎯
Skills化の効果
  • 熟練ノウハウの外部化: 暗黙知を明示的な手順として文書化
  • 生成AI注入可能: 構造化されたプロンプトとして活用
  • 品質の標準化: 属人性を排除した一定品質の担保
  • 継承・教育支援: 新人・若手への技術継承を効率化
